### 📋 Decisiones Técnicas Tomadas
```
1. TESTING STACK: Vitest + Playwright + fetch
   Vitest: Unit + Integration (Node-based, happy-dom, v8 coverage)
   Playwright: E2E (5 browsers, sequential execution)
   fetch: Real HTTP requests (no supertest dependency)
   
2. MOCKING STRATEGY: Hybrid (full mock unit, partial integration, real E2E)
   Unit: Mock Prisma completely (vi.mock), mock NextAuth (null)
   Integration: Mock Prisma with constraints, real JWT auth, real API
   E2E: NO mocking, full stack (real DB, real API, real browser)
   
3. E2E PARALLELIZATION: Sequential (fullyParallel: false)
   Rationale: Prevents race conditions in shared test database
   Trade-off: ~60sec vs ~20sec parallel, but deterministic
   CRÍTICO: 2-browser race condition test still works (1 worker, 2 contexts)
   
4. ENVIRONMENT: happy-dom vs jsdom
   Decision: happy-dom ✅ (3-5x faster, sufficient for unit tests)
   Alternative: jsdom (more complete but slower, CSS/layout not needed)
   
5. COVERAGE THRESHOLD: 80% minimum + 100% critical paths
   lib/services/** — 100% (crearReserva, cancelarReserva, auth logic)
   lib/utils/horarios.ts — 100% (hayConflicto, detectarConflicto)
   lib/validations/** — 100% (Zod schemas)
   Rationale: Balance entre covertura exhaustiva y feedback rápido

6. RACE CONDITION VALIDATION: 3-LAYER DEFENSIVA
   Layer 1 (DB): PostgreSQL UNIQUE constraint (salonId, fecha, horaInicio, horaFin)
   Layer 2 (ORM): Prisma $transaction + SELECT...FOR UPDATE
   Layer 3 (App): hayConflicto check pre-insert + React Query auto-refresh
   
7. ADYACENTES: Correctness Critical Decision
   fin1 < horaInicio2 NO es conflicto (permite cambio de profesor)
   Ejemplo: [10:00-11:00] + [11:00-12:00] = ambas PERMITIDAS
   Validado en 5+ test cases específicos
   
Riesgos identificados: 3 gaps documentados (load test, WebSocket, auditoría)
Prioridades: Load test (media, Phase 8), WebSocket (RFC futura), auditoría (Phase 8)
```
